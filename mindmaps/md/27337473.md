# 《JavaScript设计模式与开发实践》· L3 研究

- 〇 书籍元数据【原文】
  - 作者：曾探（2007 年毕业于吉林大学软件学院，腾讯 AlloyTeam 前端团队高级工程师）
  - 出版社：人民邮电出版社｜2015 年 5 月（图灵原创）｜ISBN 978-7-115-38888-9
  - 篇幅：全书三部分、22 章，系统讲解 JavaScript 中 16 个设计模式 + 面向对象设计原则
  - 分类：计算机-编程设计｜定位：面向初/中/高级 Web 前端，尤适向架构师晋级的中高级程序员
  - 学科定调【待你确认】：主学科=软件工程/设计模式（JavaScript 实现）；辅助学科=前端工程、面向对象与函数式编程
  - 读者定位：JavaScript 开发者、前端工程师、想理解"如何写可维护可复用代码"的工程师

- ① 前置元评估（止损门）【推导｜基于简介+目录】
  - 核心创见：不照搬 GoF《设计模式》的 Java/C++ 语境，而是"在尊重《设计模式》原意的同时，针对 JavaScript 语言特性"（动态类型、函数即对象、鸭子类型）重新实现 16 个模式，并用"一步步完善示例代码"由浅入深地讲——把学院派模式转成前端工程师能直接落地的代码。
  - 判定：✅ 继续（值得研究级拆解）——它是技术书，L3 价值在于以它为锚点，向外接入 GoF 经典、重构与领域驱动，并批判"模式滥用"。
  - 绑定条件：L3 研究 → 维度4 执行全部三层（审稿人+对立学者/流派+学科史），并补延伸书单与文献对照。

- ② 五维深度拆解
  - 维度1·背景 Why【原文｜内容提要】
    - 写作动机："本书根据 JavaScript 语言的特性，全面总结了实际工作中常用的设计模式"，"书中所有示例均来自作者长期的开发实践，与实际开发密切相关"。
    - 真实背景：2015 年前后前端工程化兴起，jQuery 时代零散脚本转向大型 SPA，工程师急需"结构化和可维护的代码"方法；但 GoF 原书以静态类型语言为例，JS 开发者难直接迁移。
    - 定位矛盾：它既讲"模式"，又反复强调"开放—封闭""最少知识"，避免为模式而模式——对初学者易误读为"背下 16 个模式就能写好代码"。
  - 维度2·逻辑架构 What/How【原文｜真实目录】
    - 第一部分 基础知识：第1章 面向对象的 JavaScript（1.1 动态类型语言和鸭子类型 / 1.2 多态 / 1.3 封装 / 1.4 原型模式与基于原型的 JS 对象系统）；第2章 this、call 和 apply；第3章 闭包和高阶函数
    - 第二部分 设计模式（16 个）：第4章 单例 / 第5章 策略 / 第6章 代理 / 第7章 迭代器 / 第8章 发布—订阅 / 第9章 命令 / 第10章 组合 / 第11章 模板方法 / 第12章 享元 / 第13章 职责链 / 第14章 中介者 / 第15章 装饰者 / 第16章 状态 / 第17章 适配器
    - 第三部分 设计原则和编程技巧：第18章 单一职责原则 / 第19章 最少知识原则 / 第20章 开放—封闭原则 / 第21章 接口和面向接口编程 / 第22章 代码重构
    - 【贯穿全书锚点】"计算奖金"案例从第一部分（if-else 坏味道）到第五章策略模式重构，串起"变化与不变分离"的主线，是全书的示范母题。
  - 维度3·知识细节 Top 10 核心概念【原文｜章节+原话/代码】
    - 1) 策略模式定义（原话）："定义一系列的算法，把它们一个个封装起来，并且使它们可以互相替换。"（第5章）
    - 2) 原话·设计模式的总主题："将不变的部分和变化的部分隔开是每个设计模式的主题，策略模式也不例外，策略模式的目的就是将算法的使用与算法的实现分离开来。"
    - 3) 计算奖金真实代码（带数字）：`performanceS=salary*4; performanceA=salary*3; performanceB=salary*2`；测试 `bonus.setSalary(10000); setStrategy(new performanceS())` → 输出 40000；`setStrategy(new performanceA())` → 30000；`calculateBonus('S',20000)` → 80000【原文｜第5章】。
    - 4) JS 版策略（函数即对象，原话）："在 JavaScript 语言中，函数也是对象，所以更简单和直接的做法是把 strategy 直接定义为函数"——于是 `var strategies={"S":fn,"A":fn,"B":fn}; calculateBonus=function(level,salary){return strategies[level](salary);}`【原文】。
    - 5) 鸭子类型（第1.1）："动态类型语言和鸭子类型"——只关注对象"有什么行为"而非"是什么类型"，是 JS 多态的基础。
    - 6) 发布-订阅真实例子（第8章）："真实的例子—网站登录"——用自定义事件解耦模块间通信；"必须先订阅再发布吗"讨论异步边界【原文】。
    - 7) 代理模式（第6章）："虚拟代理实现图片预加载"；"第一个例子—小明追 MM 的故事"——保护代理与虚拟代理；"代理和本体接口的一致性"是关键【原文】。
    - 8) 命令模式（第9章）："菜单程序""撤销命令""撤消和重做""命令队列""宏命令"——把"请求"封装成对象以便排队/撤销【原文】。
    - 9) 闭包与高阶函数（第3章）："闭包""高阶函数"被明确列为实现多数模式的 JS 基石工具【原文｜第一部分】。
    - 10) 开放—封闭原则（第20章）：软件实体对扩展开放、对修改封闭；策略模式"提供了对开放—封闭原则的完美支持"【原文｜第5/20章】。
  - 维度4·批判性逻辑 L3（三层全开）【原文｜文本+外援】
    - 子层A·审稿人视角（本书内部的逻辑漏洞）【推导｜基于书评与局限自陈】
      - 漏洞1·出版年份的时效性：2015 年成书，彼时 ES5 为主；ES6+（class、Symbol、Proxy 原生、async）已大幅改变模式写法，部分示例用"模拟 OO"如今可直接用语言特性，读者需甄别。
      - 漏洞2·"模式清单"的诱导：16 个模式逐个讲，初学者易陷入"哪里都想套模式"，而作者虽在第三部分讲原则，但前两部分仍偏"展示"而非"何时不用"。
      - 漏洞3·只谈前端单体示例：示例多聚焦表单校验、动画、图片预加载，缺大型应用（状态管理、微服务前端）的端到端整合，难直接迁移到复杂工程。
      - 漏洞4·"所有示例来自长期实践"但未给可运行仓库/测试，二手读者难验证代码是否仍正确。
    - 子层B·对立学者视角（其他范式如何反驳/补充本书）【外援｜外部学派】
      - GoF 原教旨（Gamma 等《设计模式》）：本书用"函数即对象"弱化类层级，GoF 重静态类型下的类结构；经典派会认为 JS 版"策略"已近乎语言内建，不必称模式。
      - 简约/反模式派（Kevlin Henney、避免模式滥用）：过度套模式增加间接层；现代前端更推崇"组合优于继承""小函数+纯函数"，本书的部分 OO 模拟显得冗余。
      - 现代框架范式（React Hooks / 函数式）：状态、订阅等已被框架内建，本书手搓发布-订阅在 Redux/Context 时代必要性下降。
    - 子层C·学科史视角（这套知识在学科脉络里的位置与演变）【外援｜软件工程史】
      - 1970s 结构化编程 → 1980s 面向对象（Smalltalk/C++）→ 1994 GoF《设计模式》确立 23 个模式的语言无关范本。
      - JS（1995 诞生）早期被当"玩具脚本"；2005 AJAX + 2009 Node.js + 2010s SPA 让 JS 工程化，催生"针对 JS 讲模式"的本土化需求（本书即产物）。
      - 2015 ES6 与随后框架（React/Vue）推动"用语言特性替代手写模式"，设计模式从"必学清单"转为"理解思想即可"，本书是这一转折期的典型教材。
  - 维度5·连接与迁移【推导】
    - → 其他动态语言：Python/Ruby 的鸭子类型、函数即对象，与本书策略/命令思路可互通。
    - → 架构：发布-订阅、中介者思想是事件总线、Redux、消息队列的雏形，可迁移到系统解耦。
    - → 通用工程：开放—封闭、单一职责是任何"可维护系统"的底层原则，超出前端。

- ③ 概念树（以"变化与不变分离"为根，长出策略模式）【原文｜第1、3、5章】
  - 根：设计模式的总主题——"将不变的部分和变化的部分隔开"
    - 前提：代码中存在"会变的算法/行为"与"不变的调用流程"
    - ↓ 推导/演变
      - 坏味道：用 if-else 按等级分支计算奖金（S/A/B 耦合在 Context）
      - 工具：JS 的函数即对象 + 闭包/高阶函数（第1、3章铺垫）
      - 重构：把 S/A/B 计算封装为策略对象/函数，Context 只委托 `strategy.calculate(salary)`
    - ↓ 结论/应用
      - 结论：消除条件分支、易扩展（加 C 级不改动旧逻辑）、符合开放—封闭
      - 应用：缓动动画、表单校验、更广义的"算法"切换（第5.4–5.6）

- ④ 反常识预警 🚨（从书里长出，带案例）
  - 🚨 反常识1：在 JavaScript 里，"策略模式"其实可以没有"类"。案例：本书直言"函数也是对象，所以更简单和直接的做法是把 strategy 直接定义为函数"，用 `strategies={"S":fn,"A":fn}` 字典即完成策略切换——经典 GoF 的 OO 骨架在 JS 中坍缩成一张函数表【原文｜第5章】。
  - 🚨 反常识2："知道一个东西的类型"不重要，"它会做什么"才重要。案例：第1章"动态类型语言和鸭子类型"——JS 多态不看类的继承关系，只看对象有没有对应行为，这与 Java/C++ 必须实现接口才能替换截然相反【原文｜第1.1】。
  - 🚨 反常识3：加一个绩效等级，竟"不用改任何旧逻辑"。案例：计算奖金从 if-else 重构为策略后，新增 C 级只需往 `strategies` 里加一项，Context 完全不动——"对扩展开放、对修改封闭"在真实代码里长这样【原文｜第5/20章】。

- ⑥ 跨界启发与落地验证【推导+外援】
  - 行动清单（5 条）：
    - 1. 遇到 if-else 按类型/等级分支，先问"这是变化部分吗"，是就用策略表替换【原文｜第5章】。
    - 2. 写模块时，用发布-订阅/事件解耦，避免互相直接依赖【原文｜第8章】。
    - 3. 抽象请求为命令对象，给"撤销/重做/队列"留接口【原文｜第9章】。
    - 4. 新增功能时优先"加策略/加分支对象"，不改动已有调用流程（开放—封闭）【原文｜第20章】。
    - 5. 用鸭子类型思维评审接口：看"能做什么"而非"是什么类"【原文｜第1.1】。
  - 验证复盘设问（量化指标）：
    - 能否把一段现有 if-else 分支重构成策略函数表，且新增分支不改动旧代码？→ 能=真懂策略模式。
    - 拿到一个耦合严重的模块，能否指出可用发布-订阅/中介者解耦？→ 能=内化解耦思想。
    - 解释"为何 JS 的策略模式可以没有类"时，能否用"函数也是对象"说清？→ 能=接住 L3 语言特性延伸。

  - 维度3·知识细节 Top 10 核心概念【原文｜章节+技术定义】
    - 1) loader 本质公式（原话性定义）：「每个 loader 本质上都是一个函数，output = loader(input)」；链式 `output = loaderA(loaderB(loaderC(input)))`，如 `style标签 = style-loader(css-loader(sass-loader(SCSS)))`【原文｜第4章，读者摘录】
    - 2) loader 执行顺序铁律：「webpack 打包时按数组从后往前将资源交给 loader，因此要把最后生效的放前面」；loader 四类 pre/inline/normal/post，用 `enforce` 指定【原文｜第4章】
    - 3) `exclude` 优先级高于 `include`；`resource` 是被加载模块、`issuer` 是加载者，用于精确定位规则作用域【原文｜第4章】
    - 4) 「一切皆模块」：Webpack 只认识 JS，其他资源须由 loader 转译为可接收形式后再继续——loader 做的是预处理器工作【原文｜第4章开篇】
    - 5) 模块系统双轨：CommonJS（动态、值复制、运行时）vs ES6 Module（静态、动态映射、编译期）——理解 tree shaking 必须分清二者【原文｜第2章】
    - 6) 代码分片演进：`CommonsChunkPlugin`（旧，有不足）→ `optimization.SplitChunks`（新），配合 hash 实现「长效缓存」【原文｜第6章】
    - 7) 样式分离：`extract-text-webpack-plugin`（旧）→ `mini-css-extract-plugin`（第2版推荐）【原文｜第5章】
    - 8) 打包优化四板斧：HappyPack 多进程／缩小作用域（exclude/include）／DllPlugin 动态链接库预编译／tree shaking 去死代码【原文｜第8章】
    - 9) HMR 原理：模块热替换，开发时局部更新而非整页刷新，含 `module.hot` API【原文｜第9章】
    - 10) 打包机制内核：第10章剖开 `Compiler`（总调度）→ `Compilation`（单次构建）→ `Resolver`/`ModuleFactory`/`Parser`→ 模板渲染，插件通过 `Tapable` 钩子协同【原文｜第10章】
    - 原话补遗：第12章直言打包工具趋势＝「性能与通用性」「配置极小化与工程标准化」「WebAssembly」；Vite 被定位为「取长补短」的新方向
  - 维度4·批判性逻辑 L3（三层全开）【原文｜书评+外援】
    - 子层A·审稿人视角（本书内部的逻辑漏洞）【推导｜基于书评与局限自陈】
      - 漏洞1·「实战」名不副实于进阶段：前 9 章偏配置罗列，第 10 章原理突兀加深，初学者易在 loader/插件机制断层；有读者笔记直言「哎，垃圾书？」式落差（指部分章节跳跃）
      - 漏洞2·版本时效性硬伤：第2版（2022-03）仍大篇幅讲 `CommonsChunkPlugin`/旧插件作为「对比」，而 2024+ 生态已全面 `SplitChunks`；前端构建在书出版后即向 Vite 快速迁移
      - 漏洞3·优化手段部分过时：HappyPack 在多进程上已被 `thread-loader`/原生并行取代；DllPlugin 预编译在现代 Webpack 收益下降——书未充分预警「哪些仍常用、哪些已 deprecated」
      - 漏洞4·缺「为什么这样设计」的取舍讨论：对配置项给「怎么做」多、「何时不该做」少，新手易过度优化（如盲目开 DllPlugin 反而变慢）
    - 子层B·对立学者视角（其他范式如何反驳本书）【外援｜外部学派/竞品】
      - 无打包（No-bundle）派·Vite：基于原生 ESM 的 dev server，跳过 dev 期打包，启动秒级；本书第12章虽提及却未动摇「Webpack 中心」叙事——Vite 用户会认为「先学 Webpack 全量配置是沉没成本」
      - 库构建派·Rollup：以 tree-shaking 干净、输出格式多选著称，更适合 npm 库；本书第11章用 Webpack 跑 React/Vue，但库场景 Rollup 更优
      - 零配置派·Parcel/esbuild：Parcel「零配置」、esbuild（Go 编写）速度数量级领先；它们直接反驳本书「配置即能力」的繁复哲学
      - 框架内置派：Create React App / Vue CLI / Vite 脚手架把 Webpack 封装到底层，多数业务开发者「不需再读本书」即可开发——挑战本书的「必读性」
    - 子层C·学科史视角（这套知识在学科脉络里的位置与演变）【外援｜前端工程史】
      - 构建工具代际：Grunt/Gulp（任务流 runner）→ Webpack（模块 bundler 一统江湖，2014+）→ Rollup（库）/ Parcel（零配置）/ esbuild（速度）/ Vite（ESM dev，2021+）
      - 模块标准之争：AMD/UMD 过渡 → CommonJS（Node）与 ES6 Module（浏览器原生）双轨，本书第2章正是这一脉络的教材化
      - 优化范式迁移：从「多进程 HappyPack」到「原生并行+持久缓存」，从「DllPlugin 预编译」到「模块联邦/原生 ESM」，性能瓶颈的解决方式随引擎升级而换代
      - WebAssembly 入局：第12章把 WASM 列为趋势，意味着构建工具本身正用 Rust/Go 重写（esbuild/SWC），JS 写的 Webpack 在速度维度长期承压
  - 维度5·连接与迁移【推导】

- ③ 概念树（以「Webpack 构建」为根）【原文｜全书】
  - 根：源码 + 资源 → Webpack → 打包产物（bundle）
    - 输入层
      - 模块系统：CommonJS / ES6 Module（静态优先→利于 tree shaking）
      - 资源入口：context / entry / 多入口
    - 转译层（loader·一切皆模块）
      - JS：babel-loader / ts-loader
      - 样式：sass/less → PostCSS → CSS Modules → mini-css-extract
      - 其他：file/url/html/handlebars；可自定义 loader（output=loader(input)）
    - 组织层
      - 代码分片：SplitChunks / 异步加载 / 长效缓存（hash）
      - 插件：Tapable 钩子 → Compiler/Compilation 协同
    - 产出层
      - 生产：production 模式 / source-map / 压缩 / 缓存 / 体积监控
      - 优化：HappyPack / 缩小作用域 / DllPlugin / tree shaking
      - 开发：HMR / 效率插件
    - 实战与替代
      - 案例：React / Vue
      - 替代：Rollup / Parcel / esbuild / Vite（趋势：ESM/零配置/WASM）

- ④ 反常识预警 🚨（从书里长出，带案例）
  - 🚨 反常识1：loader 数组「从后往前」执行，最后生效的反而要写在最前面。案例：`use: ['style-loader','css-loader','sass-loader']` 实际执行顺序是 sass→css→style，新手常因顺序写错而报错【原文｜第4章】
  - 🚨 反常识2：想让 tree shaking 生效，反而要「少动」CommonJS、多用 ES6 Module。案例：第2章讲 CommonJS 是「值复制、动态」，ES6 Module 是「静态、动态映射」；只有静态的 ES Module 才能被编译期摇树，混用 CJS 会失效【原文｜第2章】
  - 🚨 反常识3：书自己承认 Webpack 不是终点——Vite/Rollup/esbuild 各有压倒性场景。案例：第12章直言 Vite「取长补短」、esbuild「速度碾压」、趋势走向「配置极小化与 WebAssembly」，颠覆「学完 Webpack 就够用」的错觉【原文｜第12章】

- ② 五维深度拆解
  - 维度1·背景 Why【原文｜前言/读者评】

  - 维度3·知识细节 Top 10 核心概念【原文｜章节+原话摘录】
    - 1) 权衡的艺术（全书记忆锚）：命令式关注"过程(how)"，声明式关注"结果(what)"；声明式并不天然更快，它把"找差异"的成本藏起来而非消除【第1章】
    - 2) 虚拟 DOM 性能真相：「虚拟 DOM 的性能到底如何」一节指出——虚拟 DOM 并非总比原生 DOM 快，它赢在"可维护性"与"可接受性能"的折中；小页面直接 innerHTML 可能更快【第1.3节】
    - 3) 运行时 vs 编译时：「运行时和编译时」一节给出三类框架定位——纯运行时(手写 h)、纯编译时(Svelte)、通用(运行时+编译时，Vue 选此)【第1.4节】
    - 4) 响应式三件套（原话式定义）：「副作用函数就是会对外部造成影响的函数，比如修改了全局变量」；「响应式数据：修改了某个值的时候，某个会读取该值的副作用函数能够自动重新执行」【第4.1节】
    - 5) 桶(bucket)依赖收集：用 Set 存 effect，读取时 track、设置时 trigger；Vue2 用 Object.defineProperty，Vue3 用 Proxy【第4.2–5.2节】。原话：「在 ES2015 之前，只能通过 Object.defineProperty……这是 Vue.js 2 采用的方式；在 ES2015+ 中，可以使用代理对象 Proxy……这是 Vue.js 3 采用的方式」
    - 6) 嵌套 effect 与 effect 栈、避免无限递归：effect 嵌套（渲染函数包渲染函数）需栈管理；设置值时若又在 effect 内读取自身会无限递归，需 activeEffect 守卫【第4.5–4.6节】
    - 7) computed 与 lazy、watch：computed 基于 lazy effect + 调度；watch 复用同样的依赖收集机制，区分"立即执行"与"回调执行时机"【第4.8–4.10节】
    - 8) ref 解决原始值响应式：原始值(string/number/boolean)无法被 Proxy 代理，用 { value } 对象包裹；引伸出"响应丢失"与"自动脱 ref"问题【第6章】
    - 9) 三大 Diff 算法演进：简单 Diff（key 复用·DOM 移动）·双端 Diff（头尾指针比较）·快速 Diff（最长递增子序列 + 前后置相同元素，Vue 3 采用）【第9–11章】
    - 10) 编译优化三板斧：Block + PatchFlags（只比动态节点）·静态提升·预字符串化，都是"编译时分析模板、运行时只更新必要部分"的体现【第17章】
  - 维度4·批判性逻辑 L3（三层全开）【原文｜书评+外援】
    - 子层A·审稿人视角（本书内部的逻辑漏洞）【推导｜基于书评与局限自陈】
      - 漏洞1·重设计思想、轻逐行工程：名为"设计与实现"，实则"设计"占比远高于"实现"；想啃具体工程细节的读者需另看《Vue.js 技术内幕》类书
      - 漏洞2·只以 Vue 3 为唯一坐标系：全篇不对比 React/Solid/Svelte 的取舍，容易把 Vue 的"权衡"误读成"唯一真理"
      - 漏洞3·示例高度简化：为讲清原理的 demo 直接照搬到生产会踩坑（调度器边界、错误处理边界需自行加固）
      - 漏洞4·未覆盖生态：Vue Router、Pinia/状态管理、Vite 工程化几乎未涉，读者会"懂原理不会搭项目"
    - 子层B·对立学者视角（其他范式如何反驳本书）【外援｜前端框架范式】
      - React 心智模型（Meta/丹·阿布拉莫夫）：本书的"模板+运行时响应式"是 Vue 路线；React 走"JSX 即 UI + 单向数据流 + Fiber 调度"，主张"UI 是函数对状态的映射"，对"响应式依赖自动收集"持怀疑，认为显式状态更可预测
      - SolidJS（Ryan Carniato）：论证"细粒度响应式 > 虚拟 DOM Diff"——编译期把状态直接绑定到 DOM 节点，运行时零 diff，挑战本书"虚拟 DOM 是合理折中"的默认立场
      - Svelte（Rich Harris）：走"纯编译时"，把组件编译成命令式更新代码，质疑"运行时框架"的必要性，与本书第1章"通用(运行时+编译时)"选择形成对照
    - 子层C·学科史视角（这套知识在学科脉络里的位置与演变）【外援｜前端框架史】
      - 1990s jQuery 时代：命令式 DOM 操作独占；2009 AngularJS 引入"数据绑定(MVVM)"，开创响应式雏形
      - 2013 React 提出"虚拟 DOM + 声明式"；2014 Vue 1/2 用 Object.defineProperty 做响应式；2019 Vue 3 切换 Proxy+编译时优化，正本书所记
      - 响应式概念史：Knockout.js 的 observable(2009) → Vue 2 的 Object.defineProperty → Vue 3 的 Proxy → Solid 的 signal，十年内"依赖收集"从运行时走向编译期
  - 维度5·连接与迁移【推导】
    - → 框架设计：把"性能 vs 可维护性""运行时 vs 编译时"的权衡框架，迁移到自研组件库/工具的架构决策
    - → 编译优化：理解 Block/PatchFlags 后，写模板时减少动态绑定可被动获得性能提升

- ③ 概念树（以"第4章最小响应式"为根）【原文｜第一篇/第二篇】
  - 根：数据变化 → 视图自动更新（响应式）
    - 投入：副作用函数 effect 读取数据
    - 收集：读取时 track 把 effect 放进 bucket(Set)
    - 触发：设置时 trigger 取出 bucket 里的 effect 重新执行
    - ↓ 长出全部响应能力
      - 嵌套与调度：effect 栈管理嵌套；scheduler 去重/异步批量（避免抖动）
      - computed：lazy effect + 脏检查，依赖变动才重算
      - watch：复用依赖收集，在 trigger 后异步/同步执行回调
    - ↓ 与渲染器结合
      - 组件 render 函数本身是一个 effect，数据变 → 触发重新渲染 → 经快速 Diff 只更新动态节点
    - ↓ 勾稽关系
      - 响应式(第4–6章) ↔ 渲染器(第7–11章) ↔ 编译器(第15–17章) 三者通过"effect 触发更新"互相验证，任一环断了视图就不更新

- ④ 反常识预警 🚨（从书里长出，带案例）
  - 🚨 反常识1：虚拟 DOM 并不总比原生 DOM 快——它只是"可维护性"与"可接受性能"的折中，小页面/极简更新直接 innerHTML 可能更快【原文｜第1.3节】。案例：频繁更新大列表时虚拟 DOM 的 diff 开销可能反超原生，需用 key/编译优化兜底
  - 🚨 反常识2：声明式（如 Vue 模板）性能"天生"不优于命令式——它把"找差异"的成本藏起来，不是消除了【原文｜第1章】。案例：以为"写模板就快"是错觉，性能好坏取决于编译时优化是否生效
  - 🚨 反常识3：原始值(string/number)无法被 Proxy 代理，所以 Vue 3 要"多此一举"引入 ref 包裹——你以为的"数据"其实被包了一层对象【原文｜第6章】。案例：新手写 `let count = 0;` 直接改 `count` 不会触发更新，必须 `ref(0).value`，这是响应式系统对 JS 语言限制的真实妥协

- ⑤ 学科差异化结构化输出（软件工程·框架设计视角）【推导】
  - 设计主轴：权衡(命令式/声明式) → 核心要素(体积/TS/错误) → 模块协作(响应/渲染/编译)
  - 性能闭环：编译时标记动态(PatchFlags) → 运行时只比动态(快速Diff) → Tree-Shaking 砍体积
  - 排雷三连：effect 未收集→视图不更新；调度未去重→重复渲染；响应式数据被意外替换→断链

# 《你不知道的JavaScript（上卷）》· L3 研究

- 〇 书籍元数据【原文】
  - 作者：凯尔·辛普森（Kyle Simpson），图灵程序设计丛书，赵望野、梁杰译
  - 出版社：人民邮电出版社｜2015 年 4 月（中文版）
  - 页数：195 页｜ISBN 9787115385734｜定价 49 元｜开本 16 开
  - 学科定调【待你确认】：主学科=计算机科学/编程语言（JavaScript）；辅助学科=编译原理、软件工程
  - 读者定位：前端开发者、有一定 JS 经验但想理解底层机制者（"不求甚解者迎难而上"）

- ① 前置元评估（止损门）【推导｜基于简介+书评】
  - 核心创见：JS 的"复杂精妙概念（作用域、闭包、this、原型）才是语言精髓"，主张"真正理解机制而非记忆模式"，系统拆解 LHS/RHS、词法作用域、闭包、四条 this 绑定规则、原型委托（OLOO）
  - 判定：✅ 继续（值得研究级拆解）——它是前端"懂原理"运动的代表，L3 价值在机制化认知与对"反 class/委托优先"立场的三层批判
  - 绑定条件：L3 研究 → 维度4 执行全部三层（审稿人+对立学者+学科史），并补延伸书单与文献对照

- ② 五维深度拆解
  - 维度1·背景 Why【原文｜图书简介】
    - 写作动机："很多人对 JavaScript 这门语言的印象都是简单易学…殊不知，这些复杂精妙的概念才是语言的精髓，即使经验丰富的开发人员，如果没有认真学习也无法真正理解"
    - 方法立场：作者反对"因恐惧而避开特性"，主张"理解 JavaScript，而不是记忆模式"
  - 维度2·逻辑架构 What/How【原文｜目录】
    - 前言
    - 第一部分 作用域和闭包：第1章 作用域是什么（编译原理/理解作用域/作用域嵌套/异常）｜第2章 词法作用域（词法阶段/欺骗词法 eval·with·性能）｜第3章 函数作用域和块作用域（函数作用域/隐藏内部实现/IIFE/块作用域 let·const）｜第4章 提升（先有鸡还是先有蛋/编译器/函数优先）｜第5章 作用域闭包（启示/实质/循环和闭包/模块）｜附录A 动态作用域 B 块作用域替代 C this 词法 D 致谢

    - 【贯穿全书锚点】循环+闭包经典陷阱（for(var i)→打印 3,3,3；用 let 或 IIFE 解决）；模块模式（闭包封装私有状态）；this 四条绑定优先级 new>显式>隐式>默认
  - 维度3·知识细节 Top 10 核心概念【原文｜章节+原话摘录】
    - 1) 作用域定义（原话）："作用域是一套规则，用于确定在何处以及如何查找变量（标识符）"
    - 2) LHS/RHS 查询：赋值→LHS，取值→RHS；不成功 RHS 抛 ReferenceError，非严格模式下不成功 LHS 隐式创建全局变量【原文】
    - 3) 编译三步走：JS 引擎先分词/词法分析→解析(生成 AST)→代码生成；`var a = 2` 被分解为"声明(提前)"与"赋值"两步，这就是"提升"的根源【原文】
    - 4) 词法作用域（原话）："词法作用域意味着作用域是由书写代码时函数声明的位置来决定的"【原文】
    - 5) 闭包定义（原话）："当函数可以记住并访问它所在的词法作用域，即使是在当前词法作用域之外执行，这就产生了闭包"【原文｜与《语言精粹》对读】
    - 6) 循环+闭包陷阱（案例数字）：`for(var i=0;i<3;i++) setTimeout(()=>console.log(i))` 打印 3,3,3（共享同一作用域）；解决靠 let 块作用域或 IIFE【原文】
    - 7) this 本质（原话）："this 是在运行时进行绑定的，并不是在编写时绑定…它的上下文取决于函数调用时的各种条件…只取决于函数的调用方式"【原文】
    - 8) this 四条绑定规则及优先级（原话归纳）：new 绑定 > 显式绑定(call/apply/bind) > 隐式绑定 > 默认绑定（严格模式 undefined，否则全局对象）【原文】
    - 9) 隐式绑定丢失（案例）：对象方法赋值给变量再调用会变成默认绑定——`const fn = obj.say; fn()` 打印 undefined（非严格），"这正是 bind 必须存在的理由"【原文】
    - 10) 行为委托 OLOO（原话立场）：原型本质是"对象关联/委托"，而非模拟类继承；作者在第六章提倡用委托而非 class【原文】
  - 维度4·批判性逻辑 L3（三层全开）【原文｜书评+外援】
    - 子层A·审稿人视角（本书内部的逻辑漏洞）【推导｜基于书评与局限自陈】
      - 漏洞1·"理解优于记忆"理想化：对框架(React/Vue)主导、交付节奏快的团队，深挖作用域/原型性价比有限，多数业务用不到 OLOO
      - 漏洞2·对 ES6+ 滞后：上卷写于 ES5 时代，class、模块、箭头函数、Proxy 等仅附录补丁，"this 词法"(箭头函数)只是补救
      - 漏洞3·"反 class"立场偏激：第6章贬低 class/模拟类，但 ES6 class 已成主流，委托范式并非银弹
      - 漏洞4·抽象层级跳跃：从编译原理直接跳到哲学化"为什么用 this"，初学者易迷失
    - 子层B·对立学者视角（其他范式如何反驳本书）【外援｜外部学派】
      - 道格拉斯·克罗克福德《JS 语言精粹》：主张"用子集、避开糟粕（this 陷阱、with、eval）"，与辛普森"理解而非恐惧"相反——一个避坑，一个讲坑为何存在
      - TC39/现代标准：ES6+ 引入 class、模块、块作用域，事实上部分采纳了辛普森担心的"模拟类"路径，说明工程界选择了语法糖而非纯委托
      - 类型阵营（TypeScript/Flow）：认为动态 this、原型歧义正是 JS 痛点，解决靠静态类型而非"更理解原型"
    - 子层C·学科史视角（这套知识在学科脉络里的位置与演变）【外援｜语言史】
      - 词法作用域/闭包源自 Scheme/Lisp 的函数式传统（Sussman《SICP》），JS 设计借自 Scheme + Self(原型)
      - 原型继承源自 Self 语言(1980s)，辛普森的 OLOO 是对"用原型模拟类"的回拨
      - 从《JS 权威指南》(Flanagan) 的"大全式"到辛普森"机制深挖式"，反映 JS 社区从"会用"到"懂原理"的成熟
  - 维度5·连接与迁移【推导】
    - → 工程：用闭包实现模块私有化、用委托替代继承减少原型链混乱
    - → 面试/教学：四条 this 绑定+优先级是高频考点，可系统作答
    - → 跨界：LHS/RHS"查询模型"、词法作用域"定义时确定"可类比语言学/编译器原理

- ③ 概念树（以"作用域"为根）【原文｜第一部分】
  - 根：JS 引擎如何查找变量（作用域=规则）
    - 编译：分词 → 解析(AST) → 代码生成（var a=2 拆两步）
    - 查询：LHS(赋值)/RHS(取值)，逐级向上直到全局
    - 词法作用域：由声明位置决定（eval/with 欺骗词法=性能代价）
    - 函数/块作用域：隐藏内部实现（IIFE / let / const）
    - 提升：声明先扫（函数优先于变量）
    - ↓ 闭包：函数记住并访问其词法作用域 → 模块模式（私有状态）
  - 第二根：this 与对象原型【原文｜第二部分】
    - this = 运行时绑定，取决于调用方式
    - 四条规则+优先级：new > 显式 > 隐式 > 默认
    - 对象：属性描述符 / [[Get]][[Put]] / 原型链
    - ↓ 原型：[[Prototype]] / 委托(OLOO) vs 类模拟

- ④ 反常识预警 🚨（从书里长出，带案例）
  - 🚨 反常识1：JS 是"编译语言"而非纯解释执行。案例：`var a = 2` 在代码执行前已被引擎分解为声明与赋值两步，这正是"提升"之谜的根源【原文｜编译原理】
  - 🚨 反常识2：函数内部的循环变量 `var i`，所有 setTimeout 都打印同一个 "3,3,3"，因为闭包共享同一作用域。案例：for 循环经典题，解决靠 let 块作用域或 IIFE【原文｜循环和闭包】
  - 🚨 反常识3：this 指向"不是函数定义位置"而是"如何被调用"，把对象方法赋值给变量再调用，this 会丢失变全局。案例：`const fn = obj.say; fn()` 打印 undefined（非严格），这正是 bind 存在的理由【原文｜隐式绑定丢失】
